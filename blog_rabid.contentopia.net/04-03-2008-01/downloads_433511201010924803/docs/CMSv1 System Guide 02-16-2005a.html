Purpose

 

Define and document the technology and products to used in implementing the PMProfessionalism Content Management System.

 

 

Introduction

 

PMProfessionalism Content Management System provides the technical means to develop and deploy Intranet based web sites such as the PMProfessionalism site in a timely manner with little overhead and minimal programming expertise.   Some aspects of the System Guide may resemble some aspects of the User Guide in terms of describing how the system works; this was unintentional as it is a requirement for those who work on the system’s code to understand how the system works.

 

 

Scope

 

This document identifies and describes the software system upon which the PMProfessionalism Content Management System was based.

 

 

Overview

 

Requirements driving this project were communicated verbally at a fairly high-level.  The primary requirement was that non-technical people be able to manage their web content with minimal programmer or I.T. support in accordance with the SCM Policies and Procedures for SBC.

 

 

System Architecture

 

Folder Structure

 

This is the folder structure for the Content Management System as it would typically be deployed for a single site configuration.  This same code could be packaged to be deployed in a manner that would lend itself to being maintained at a lower cost in terms of how updates are distributed.  One way to deploy this code and reduce the maintenance cost would be to deploy the code to a shared folder with the necessary “hooks” to allow the code to be executed in this manner and then the code would only need to be updated in a single location rather than many locations.

 

The root for the site is, in this case, is “pmprofessionalism” however the root folder could be any folder name and is identified to the system in the typical manner depending on which web server is being used.

 

There are four (4) subsystems for this system and they are, using no particular order, as follows: (a) Admin, (b) Layout, (c) Release and (d) Security.

 

There are three additional folders worth mentioning and they are as follows: (a) components, (b) images and (c) uploaded-images.  The uploaded-images folder is created dynamically at run-time as-needed whenever an image file is uploaded to the server.  The images folder contains all the static image content for the site.  The components folder tree contains all the components for the site and support the functioning of the site at run-time.

 

The remaining folders are not part of the system and can be largely ignored for the purposes of understanding how the system is architected.  The prodbackup folder was intended to be used as a backup in case the database was offline or otherwise unavailable however this folder could be removed with no loss of functionality.

 

 

Subsystem Architecture

 

All of the subsystems, which includes 100% of the GUI, uses the same base ColdFusion code to maximize the reuse of code and minimize the coding maintenance effort.  Maximizing code reuse however does tend to require slightly more code complexity than if reuse were not a consideration.

 

Code Reuse Mechanism

 

Code reuse is achieved by utilizing one central ColdFusion file from the root folder called “index.cfm”.  Each of the subsystems then in-turn create their own contexts prior to performing a <cfinclude> that references the root index.cfm file.  This technique allows each subsystem to be handled differently in terms of user authentication, this was a requirement, thus allowing a shared set of code to drive the site in a state-driven manner.  Had this reuse technique not been used it would have become necessary to create no less than three nearly identical views of the site content (Live Site, Admin and Staging) which would then have been delivered through a reusable <cfinclude> file or via a .CFC (component) or some other technique that could have resulted in even more code complexity than is apparent at this time.

 

User Authentication

 

Subsystem Security

 

Users are authenticated at run-time through the use of LDAP.  LDAP is an SBC provided system that requires a user to enter a username and password whenever a user attempts to access a web page located in an LDAP protected folder.  LDAP provides two data elements to the ColdFusion code, one of which is the username and the other is the password; the password is ignored at run-time.  The ColdFusion code make the assumption that the user must be authenticated IFF the username is provided and contains a series of non-whitespace characters.  The username provided is then used to determine which subsystems the user is allowed to use other than the Security subsystem which uses only LDAP as a means of controlling access; there is no other way to deny access to the Security subsystem other than to deny access via LDAP whereas the other subsystems are controlled via LDAP in addition to settings stored via the Security subsystem.

 

Functional Security

 

In addition to subsystem security there is also the need to grant or deny access to certain functions such as the ability to Add New Pages of Content which also grants access to the Menu Editor among others.

Content Security

 

In addition to subsystem security and functional security there is also the need to grant or deny access to individual pages of content such as those that are provided by the system and those that are provided by end-users in the form of editable HTML content.  No distinction is made between system provided content pages and end-user provided content pages in terms of determining which users may access either when it comes to editing rights.  From the Content Management System’s perspective all pages of content that are editable are subject to security settings as established by the Security subsystem.

 

User Roles

 

User Roles are established by the functionality (which subsystems can a user use), the functional security settings and the content page security settings.  There is no other formalization of “Roles” and therefore no formalized way to determine what “Role” a user plays at run-time.  It was simply faster to develop this system without the formalization of roles and in the final analysis any formalization of roles would have been redundant in actual use and may have required a common GUI through which all users would play their role which might have given rise to far more code complexity than may have been desired.  This informal determination of user roles also provides for greater end-user security because end-users are not faced with knowing about subsystems or functions that cannot access; inaccessible subsystems or inaccessible functions are not shown to the end-user at run-time.

 

1.1.1.1.1          Admin Role

 

The Admin Role is also known as the Content Management Role and is used to maintain content for the site.  Any user who has been granted access to the Admin subsystem via the Security subsystem and has access via LDAP can play the role of  Content Manager.  A Content Manager can typically edit pages of content but not add new pages of content unless that same user has been granted the specific access to the function that allows new pages of content to be added which also includes the ability to edit the menu.  Any pages of content that are added using the appropriate function will automatically inherit the security settings of the user who added the page.  In other words, all pages added by Sam are automatically made accessible to Sam without the need for any other actions to be taken by the Security Manager. 

 

1.1.1.1.1.1          Serialization of the Admin Role

 

One or more users can play the role of Content Manager as long as they are not all being granted the ability to edit the same pages of content at the same time.  The system takes no actions to serialize access to editable pages of content which means if more than one user edits the same page of content at the same time the user who last hit the save button will overwrite the user who first hit the save button and in the extreme case the page of content could become corrupt.  Additionally the database makes no attempt to utilize transaction processing nor does it serialize access to database tables which typically poses no problems for the database unless more than one user attempts to edit the same page of content at the same time. 

 

1.1.1.1.1.2          Serialization of the Menu Editor

 

Access to the menu editor is serialized and marshaled by the database in order to avoid as possibility of data becoming corrupt by more than one user editing the menu at the same time.

 

1.1.1.1.2          Release Manager Role

 

The Release Manager Role is responsible for controlling the “state” of a Release through the lifecycle of a Release.  Any user who has been granted access to the Release subsystem via the Security subsystem and has access via LDAP can play the role of  Release Manager.

 

1.1.1.1.2.1          Serialization of the Release Manager Role

 

Only one user can play the role of Release Manager at a time.  The system takes absolutely no actions to serialize access to a Release which means if more than one user attempts to change the “state” of the same Release at the same time the user who last hit the commit button will overwrite the user who first hit the commit button and in the extreme case the “state” of the Release could become corrupt and unusable.  Additionally the database makes no attempt to utilize transaction processing nor does it serialize access to database tables which typically poses no problems for the database unless more than one user attempts to change the “state” of a Release at the same time. 

 

1.1.1.1.3          Security Manager Role

 

The Security Manager Role is responsible for controlling the “accessibility” of content within a Draft Release.  Any user who has been granted access to the Security subsystem via LDAP can play the role of  Security Manager.

 

1.1.1.1.3.1          Serialization of the Security Manager Role

 

Only one user can play the role of Security Manager at a time.  The system takes absolutely no actions to serialize access to the Security Settings for a Draft Release which means if more than one user attempts to change security settings for the same Draft Release at the same time the user who last hit the save button will overwrite the user who first hit the save button and in the extreme case the security settings of the Release could become corrupt and unusable.  Additionally the database makes no attempt to utilize transaction processing nor does it serialize access to database tables which typically poses no problems for the database unless more than one user attempts to change security settings of a Draft Release at the same time. 

 

Database Architecture

 

Discussions  about the database need to be split between the Development database and the Production database. 

Development Database

 

The Development database that was used for this project was originally hosted on a SQL Server that was shared between Development and Staging.  During the development cycle of this project however the need arose to allow the Staging server to be used for parallel testing efforts while the development database was being used for development efforts.   This requirement gave rise to a set of problems that would have required the development SQL Server to be hosted separate from the staging SQL Server because had both efforts used the same database both efforts would have been changing “state” in the database for the same data objects and this would have caused an unacceptable level of chaos for both efforts.

Duplicate Parallel Database Tables

 

Because both development and staging had to be useful at the same time it became necessary to create a duplicate set of database tables (one for one) because it was not possible to get a separate SQL Server database for staging testing efforts.  Each of the database tables that was owned by “dbo” was duplicated and owned by “[ITSERVICES\ps6473]” and post-ended with the number “2”, for lack of a better mechanism.  The naming convention used for the duplicate set could have taken on any usable form however the one used seemed to be easier to discern than merely looking for a slight different between table names during development.

 

ColdFusion’s view of the Duplicate Parallel Database Tables

 

Because this duplicate set of database tables had to exist in parallel with each other the need arose within the ColdFusion code to somehow allow one set to be used for development and the other set to be used for staging testing.   The way this was handled and managed grew out of the fact that SBC sets a ColdFusion variable to define which environment the code in running in such that it was possible to use one set of tables during development and the other set during staging testing.  This meant there would be no Stored Procedures for this project and all SQL statements (T-SQL Scripts) would be dynamically created at run-time by the ColdFusion server but the problem was solved and all parties were happy.

T-SQL Scripts

 

T-SQL also known as Transact SQL is the language of SQL Server 2000.  T-SQL Scripts for this project were designed to be monolithic T-SQL Script in order to reduce the number of <CFQUERY> invocations to a minimum rather than the more typical case in which rather small SQL Statements are used with no regard to the number of <CFQUERY> invocations used.  The former requires a fair amount of programmatic logic to support the processing that runs in the database engine while the latter allows for a more forgiving architecture in which database problems are handled programmatically in a much more graceful manner than might be allowed within a T-SQL Script.

 

T-SQL Error Handling

 

SQL Server 2000 does not provide the means to trap and handle errors that occur while a T-SQL Script is being executed at run-time.  It is possible to determine error codes however errors such as syntax errors will simply cause the whole script to cease execution with no way to recover from the problem other than to blindly assume the T-SQL Script works once it reaches a workable state during development and testing.

 

1.1.1.1.4          SBC Coding Guidelines Concerning Error Handling

 

SBC Coding Guidelines states the following:

 

CFTRY/CFCATCH

 

Using CTRY/CFCATCH to capture exceptions can be useful, however, it also hides errors from the ColdFusion Application log. This creates a situation in which the server administrators have NO means of determining where a problem may lie.

 

For instance, say that a particular database server is down. If a the ColdFusion templates that connect to this database has the cfquery statements wrapped within a CFTRY/CFCATCH, the server admin will have no way of knowing that this particular database is down. As requests come in to the server to use this application the cfquery statements will attempt to connect to the database. Each cfquery statement will wait up to its maximum time to obtain a connection. ColdFusion application server has a finite number of active running threads as well as a finite number of requests that can be queued. Say that the maximum number of running requests is set to 40. Every user who attempts to access this particular application will take away one of those threads. Normally the threads (hopefully) execute in few hundred milliseconds or less; freeing a thread for another process (or one in the queue). If requests executing for this application with the database that is down take at least 30 seconds for their cfquery ODBC connections to timeout; requests will not start and finish as quickly. This means that requests for this application as well as others on the server will start to accumulate the queue. Once the queue is filled users will begin to receive “Server Busy” errors. The web administrator should know about the condition at this point due to monitoring of ColdFusion, client complaints about speed, or the ColdFusion process actually crashing and requiring a restart. Although the web administrator will be able to acknowledge that there is a problem, he/she will not be able to pinpoint which site is causing the problem due to the masking of errors facilitated by CFTRY/CFCATCH. If the errors were logged in the Application.log where they should be, the web administrator would be able to contact the siteowners and alert the DBA of the problem.

 

If you are currently using CFTRY/CFCATCH to display custom error messages rather than the more cryptic default ColdFusion error messages, you can use CFERROR to obtain similar results. In fact using CFERROR to obtain these results requires far less development time. Rather than having to add CFTRY/CFCATCH clauses to each page; you can add a single CFERROR statement to your Application.cfm.

 

The reason for this SBC Guideline is fairly straight-forward within the context of the typical ColdFusion application that must be Administered by the I.T. folks who have certain requirements. 

 

The problem with this guideline is that a more complex application such as this Content Management System needs to be more robust that the typical end-user coded application and therefore there is a need to perform programmatic error handling in order to ensure the state of the database is consistent with the functions being performed by the end-user.

 

1.1.1.1.5          Adherence to SBC Coding Guidelines Concerning Error Handling

 

In a worse-case scenario it might become possible for the T-SQL Script(s) to fail to execute in a silent manner (other than the typical entries in the ColdFusion Server Logs) with almost no way to determine why a T-SQL Script failed if the failure was caused by a syntax error in which case SQL Server would simply cease the execution of the offending script which could lead to the loss of data or the loss of “state” within the database.

 

Database Connections

 

The SBC Guidelines concerning database connections and ColdFusion is as follows:

 

12) Avoid CFQUERY within a loop (CFLOOP or CFOUTPUT with the query attribute)

 

While this guideline doesn’t specifically state one should try to avoid using many <CFQUERY> invocations it does suggest one should try to code monolithic SQL Statements.  This guideline does however suggest one should not code <CFQUERY> invocations within a loop however there are times when it is necessary to code this type of logic in order to achieve the desired results.

1.1.1.1.6          Adherence to SBC Coding Guidelines Concerning Database Connections

 

Strict adherence to the SBC Coding Guidelines concerning the use of <CFQUERY> invocations may result in rather large unwieldy T-SQL Script that are less efficient than the alternative methods for achieving the same goals. 

 

The intent of this SBC Coding Guideline is to prompt casual programmers to give some thought to how they code their logic in relation to SQL Server queries however in actual practice there is little reason to red-flag every instance of a <CFQUERY> tag within a loop.

 

1.1.1.1.7          Typical ColdFusion Apps and Database Connections

 

The typical way to handle this in a ColdFusion application is to code rather small SQL Statements that are executed using <CFQUERY> invocations while allowing the ColdFusion server to handle the programmatic logic required to manipulate the result sets.

 

Sometimes it is necessary to code <CFQUERY> tags within loops however when this is required it must be done otherwise alternate code would be created to attain better performance.

 

Even when a ColdFusion application is making many connections to a SQL Server via <CFQUERY> tags it is still possible to adjust a ColdFusion server to provide a longer time-out to a database connection which results in the connection remaining open to allow many calls to <CFQUERY> to execute without the need to create many actual database connections.

 

1.1.1.1.7.1          Cost/Benefit Analysis of Ignoring this SBC Coding Guideline

 

The cost of strict adherence to this SBC Coding Guideline might be huge monolithic T-SQL Scripts that hide some programmatic logic within the T-SQL Scripts that are then subject to the rules of execution for T-SQL Script as previously stated in this Guide.

 

The benefits of ignoring this SBC Coding Guideline might be more granular use of SQL Statements with little use of T-SQL logic and a greater ability to handle database errors such as syntax errors that might creek into the system during the post-production period when the code is being used by real end-users.  Obviously it takes a certain amount of programming skill to use <CFQUERY> tags in this manner but when it works it works well and the results can be a robust application that can roll-with-the-punches more gracefully than allowed by SBC Coding Guidelines.

1.1.1.1.8          Programmatically Violating SBC Coding Guidelines

By now the casual reader is probably “crying foul” at the thought of doing this however there are times when it is necessary to produce ColdFusion code that can violate the SBC Coding Guidelines such as the following scenario:

 

1.1.1.1.8.1          One Possible Scenario

 

Consider a database table that has a Primary Key Constraint.  The application must insert records into this table.  What happens when a user attempts to insert a record that attempts to violate the Primary Key Constraint by attempting to insert a duplicate record.  The Primary Key Constraint causes the database to enforce uniqueness within a database table by disallowing duplicate records for those columns that have the Primary Key Constraint.

 

According to the SBC Coding Guidelines it is not permissible to code ColdFusion logic that can do anything about the SQL Error that is produced when a user attempts to violate a Primary Key Constraint by attempting to insert a duplicate record.  So the typical SBC user, one assumes, would be faced with some kind of generic error page handler that might send an email to someone however the end-user would be unable to proceed to work with the system and might conclude the application is broken.  In any case, this is not the type of SQL Error the I.T. folks need be concerned about because this is not in effect a real SQL Error it is however programmatic feedback to the end-user that indicates the data needs to be changed before it can be saved to the database.  

 

It makes very little sense to code an application that fails to handle a simple Primary Key Constraint violation attempt and it can be quite frustrating for end-users to have to get help from someone just to find out their data needed to be changed in order for them to proceed with their work.  The typical novice user might not even know they can hit the browser’s back button to get back to the page that caused the error and some browsers might fail to show the page that caused the error with the originally entered data intact.  Additionally the ColdFusion code might be using Meta Tags that inhibit the caching of pages that are being generated dynamically which means the browser might not be able to show the previous page from the browser’s history.

 

1.1.1.1.8.2          How-To Programmatically Violate SBC Coding Guidelines

 

The current, as of this writing, SiteMover CodeScanner does not look for forbidden tags that are coded within <CFSCRIPT> code blocks which can be placed within .CFC files or elsewhere.

 

One can code a <CFTRY> tag which is forbidden if one uses the variant of this tag that can reside within a <CFSCRIPT> code block.  The actual details about “how” this is done can be found in numerous sources on the Internet via a Google Search or the like.

It is a fairly trivial matter to produce a code scanner that can look for a tag that looks like <CFTRY> but it is another matter entirely to produce a code scanner that can correctly detect the <CFSCRIPT> variation of this forbidden tag.  Even if such a code scanner could be produced it would always be possible to create ColdFusion code that is created dynamically at run-time which is executed at run-time without code scanner’s ability to detect forbidden code constructs.

 

The typical ColdFusion programmer might not know how to write code that can effectively bypass the SBC code scanner and this document does not specifically state how to do so other than the abstract declaration that it can be done.

 

Nonetheless there are times when it is not only desirable to bypass some SBC Coding Guidelines but not doing so might allow code to be produced that fails to allow users to actually use the system without the need to seek help from someone who will simply tell them to try to save data that is not already in the database.

 

1.1.1.1.8.3          How-To Programmatically Bypass SBC Coding Guidelines

 

The other way to bypass SBC Coding Guidelines would be to produce far more complex code that would cost more to produce and test that would perform a sanity check on data in the database prior to performing a INSERT such as in the case of the Primary Key Constraint Violation.

 

Yes, it is possible to code a T-SQL Script that can check to see if the record with the Primary Key Constraint already exists in the table and then to not perform the INSERT which means no SQL Error condition is created and the problem can be handled silently in a graceful manner.  It might only take NN extra minutes to produce the code every time this functionality is needed and ColdFusion code could be created to facilitate the process of having to programmatically check for the Primary Key Constraint Violation attempts whenever this logic might be needed.  The Better Solution that takes very little programming effort by comparison is the one that uses the <CFTRY> logic to detect and handle the error condition.

 

There are many more non-fatal SQL Error conditions that should be handled without bothering the end-user or the I.T. Support Staff other than the Primary Key Constraint Violation condition.  Foreign Key Constraints can behave in a similar manner if one attempts to delete data from tables out of sequence, granted this is a programming logic error unless the point of the application is to grant this level of control over the database contents to the end-user.

 

GUI Architecture

 

The GUI was originally produced from an HTML template of the PMProfessionalism site and consisted of traditional ColdFusion code without any JavaScript at all.  There were few requirements other than “make the site’s content editable” and “give us an SCM process”.  When the requirements drove the design down the path of requiring more than traditional means could provide the design took a sharp right-turn towards JavaScript because this was the only way available to provide the level of functionality the design team seemed to want.

 

The Menu Editor

 

One case in point is the Menu Editor which began life as a clumsy non-JavaScript interface that just could not provide the required functionality.

 

The Better Solution

 

The Better Solution for the design might have been to code the system from the other side of the coin, to ignore the content and work from the desired solution back to the original content however doing so would have taken longer and cost more to produce with possibly better results.

 

Traditional Client-Server Architecture ?

 

Along the way additional tricks and techniques became apparent there just wasn’t sufficient time to roll into the system.  Tricks that would allow JavaScript to access the database directly using a more traditional client-server architecture.  This would allow for a more granular approach to the GUI that could have allowed for a more multi-user system rather than the requirement to serialize access to certain functions and subsystems.  If JavaScript were able to access the database to determine the “state” of objects such as the Site Menu, the Menu could have been turned into a series of database objects that could have been edited directly which could have allowed multiple people to share in the process of editing the Menu with real-time live updates as each action was performed on the Menu objects in the database.

 

1.1.1.1.9          Adherence to SBC Coding Guidelines ?

 

According to SBC Coding Guidelines, <CFQUERY> tags are not allowed within loops.  A traditional client-server application requires the use of what is in essence <CFQUERY> tags within loops in order to simulate the binding of GUI elements to database objects such as is achievable when coding traditional Visual Basic applications.

 

Typically web applications exist in a “connectionless” world in which a connection to a database cannot be maintained over time.  ColdFusion can mitigate this by providing what can appear to be “persistent” connections to the database.  JavaScript can provide the means to maintain a connection to the database.  When one user clicks on a button on a web page that causes an object in the database to change another user would then see the “state” of that object change and the GUI is updated to either disable a button or show the results of the “state” change in the database which could be an updated Menu.

 

1.1.1.1.10       Client-Server Performance Cost

 

What would be the performance cost of a bunch of users all constantly hitting the ColdFusion server in what appears to be the background from the end-users perspective ?  Yes, there would be more network traffic using a client-server model along with more web server traffic and more database traffic and more activity going on behind the scenes.  But, the application would also be more robust and easier to use and appear to be a more traditional client-server application that could allow more users to interact with data objects than would otherwise be possible.

 

Traditional Non-Client-Server Architecture ?

 

Let’s consider the problem of allowing users to Edit the Site Menu for a typical web site using the traditional web based approach.

 

Every time any user wanted to change something in the Menu they would have to make a round-trip to the server and refresh their web page.  This would be simply unacceptable to the typical end-user.  Click on something then wait.  Click on something else then wait.  Click on another something then wait.  Then click on the save button and wait.

 

1.1.1.1.11       Design Trade-Offs

 

The trade-off to all this clicking and waiting was to produce a traditional JavaScript interface that requires serialization of Menu Editor access but allows end-users to have a smoother Menu editing experience.  This is how the Menu Editor was designed for this Content Management System.  The client-server model being discussed in this Guide might have been a little less smooth than the free-flowing Menu Editor we have today but it would have been more multi-user friendly.  Sending and receiving packets with a frequency of say 2 seconds per round-trip might have provided a sufficiently smooth user experience.  Even with 1000 users all using the Menu Editor the traffic load would have been no more than 1000 connections to the database, no more than 500 transactions per second to the database and no more than 500 hits per second to the ColdFusion server.  This amounts to the same level of traffic as a site that gets 1,800,000 hits per hour.  Sounds like large numbers but well within the operational limits of most busy web servers and certainly within the operational limits of most busy database servers.  How many minutes would 1000 users spend all editing their menus each day ?  Probably not that many minutes and certainly not even close to an hour each.  If performance became an issue the system could be coded to respond by increasing the delay between packet round trips however this could have been coded as part of the design to automatically adjust the delay based on server response times.  The longer the delay became the longer the delay would be made until the expected delay was equal to the requested delay.  It would be a trivial matter to create JavaScript code that could automatically adjust itself based on actual system performance such that the resulting code would tune itself to avoid flooding the server with too much traffic during peak load times.

 

System Extensibility

 

 

The software and database for this CMS were not designed to be easily extended without the talents of a sufficiently skilled ColdFusion programmer and/or DBA.  The code and the database, as delivered to Production on/about 02-14-2005, was hastily created without any real thought as to “how” either would be easily extended.  On the other hand, a sufficiently skilled ColdFusion programmer who has in-depth knowledge of JavaScript 1.2 should have no real problems extending the current system to do anything that can be done.  This is how the system was designed and implemented; constantly extending the base system to include new features using updated methods and techniques.  The same can be said about the database; a sufficiently skilled DBA should have no problems extending the database to include newer features and functionality.

 

Database Extensibility

 

The current database needs to be reworked, somewhat, to remove some of the tables that became redundant over the life-cycle of the development of this system.

 

Redundant Database Tables

 

The table named “DynamicHTMLContent” could be made to handle the functionality of the following tables: DynamicHTMLsepg_section, DynamicHTMLsepg_links, DynamicHTMLright_side, DynamicHTMLmenu, DynamicHTMLfooter and DynamicHTMLpad. 

 

Each of these tables performs the same identical functionality within the database for the system.  At the time, however, it was easier to simply create another table and effectively “firewall” the code that used the other tables and move the development process along than to combine all the tables into a single table.  Additionally, at the time, the goal was to try to limit the number of <CFQUERY> tag uses to as small a number as possible however doing so proved to be unwise.

 

The process of removing the redundant tables in favor of the DynamicHTMLContent table would be quite simple.  A ColdFusion programmer who knows how to recode T-SQL Scripts could easily modify the T-SQL Scripts that use each of the redundant tables to use the DynamicHTMLContent table instead.  A DBA would be needed to add a column to the DynamicHTMLContent table along the lines of a column named “contentType”.  Because it is necessary to extract data for a typical site layout from all of these redundant tables a column named “contentType” would make this a bit easier but the same thing could be accomplished using only the pageId column from the DynamicHTMLContent table and the PageName column from the DynamicPageManagement table.  An IN clause would then be needed to extract content from all the required content types from the one DynamicHTMLContent table however it might be easier to code a query that would extract all the data for a Release using the rid column and then use Query of Queries to extract the individual content elements from the bulk query.

 

Brief word about Query of Queries

 

In ColdFusion 6.1 MX it is possible to code a <CFQUERY> that uses a previously created Query data object in place of a Table name with the result being a no-performance-cost sub-query.  Consider a query that returns a large result set that contains 20,000 rows of data.  A <CFLOOP> could be coded in ColdFusion to process this huge result set however doing so might result in a performance problem.  A Query of Queries could be coded to produce a smaller more precise result set that is based on the larger result set that contains 20,000 rows without the need to hit the database server.  In fact, it would be possible to issue almost any number of Query of Queries against that same 20,000 row result set with nearly no cost in performance, within certain guidelines.  The typical way to handle this in ColdFusion 6.1 MX would be to issue a cached <CFQUERY> to extract a fairly large result set using a fairly simple SQL Statement and then issue Query of Queries to extract more precise result sets from the large cached query all without the need to code cumbersome T-SQL Scripts thus providing the means to handle database errors within the application code with the benefit being a more robust system that is actually easier to maintain.

 

1.1.1.1.12       Adherence to SBC Coding Guidelines

 

At the time the current SBC Coding Guidelines were written ColdFusion 6.1 MX was not in-use.  The CodeScanner used by SiteMover cannot discern between the more typical but slower <CFQUERY> within a <CFLOOP> and the faster Query of Queries with a <CFLOOP> as both appear to be identical at the tag level; the only difference being the source table name for both queries.  Therefore, gratuitous use of Query of Queries would appear to be a performance problem based solely on the SiteMover’s Report that might indicate problems with uses of <CFQUERY> within a <CFLOOP>.  This is the case with this system.

 

The SBC Coding Guidelines should be brought up to date with current ColdFusion 6.1 MX techniques and the CodeScanner should be updated or ignored.

Using the Hierarchical System Chart above you may notice (reading from top to bottom) that ColdFusion interacts with the Database to produce HTML content and JavaScript content; JavaScript content sometimes creates HTML content dynamically and there is some use of DHTML and DOM (Document Object Model).

 

The typical flow of control goes something like this:  ColdFusion file named index.cfm is executed which then creates a mixture of HTML/JavaScript/DHTML with database queries mixed in during the execution of the ColdFusion code.  After ColdFusion finishes the HTML/JavaScript/DHTML and DOM takes over and allows for user interaction until the user clicks a button or a link that fires off the ColdFusion server once again to save data and retrieve data from the database.  DOM is typically used to directly manipulate HTML objects from JavaScript such as when a drop-down list box is being updated.  DHTML is typically used to perform functions such as displaying the site menu which at this time is largely a DHTML object controlled entirely by the browser using a set of CSS definitions.

 

Client-Server JavaScript

 

When this development effort originally began, the notion of being able to manipulate database objects directly using JavaScript was a pie-in-the-sky idea. It would have been a nice technique to use that could have possibly resolved some multi-user technical issues. However the realization of this concept was not understood until this effort was too near the end of the allowed time.

 

Client-Server JavaScript Described

 

In short, Client-Server JavaScript provides the means for a static GUI to remain visible at all times meanwhile database accesses occur in the background with no break in the user experience to refresh the web page or otherwise post data to the server, or so it seems.  JavaScript and DOM are used to create HTML content and otherwise control the user experience.  HTML code can be downloaded from the server when doing so becomes appropriate.  DHTML is used to create HTML content when downloading said content from the server poses technical problems or when doing so seems more advantageous.

 

SBC Coding Guidelines

 

Client-Server JavaScript might not be a violation of the working of the SBC Coding Guidelines but implementing this technique would require a relaxation of the Guideline that mentions how often or how many <CFQUERY> tags might be used.

 

Performance Considerations

 

Client-Server JavaScript might result in more traffic to the ColdFusion server farm and likewise it may result in more traffic to the database server farm but the benefits out-weigh the apparent cost in performance.  In real terms, there are relatively few times when the JavaScript Client would be required to appear to go into a loop looking for updates from the Server.  It might only be necessary to request the status of certain GUI objects from the server such as when a GUI object receives focus prior to the user being allowed to click on a GUI object to determine if the user-click is a valid action at the moment based on the state of that object in the database. 

 

Complexity of Code

 

That sure does sound rather involved.  But what if we wanted to design a Menu Editor, for instance, that a group of users could all use at the same time to share in the experience of editing the site menu?  All the users who are viewing the Menu Editor would be able to see the same menu objects in the database.  When User-A clicks on a menu prompt to begin editing the JavaScript Client would send a request to the server to attempt to flag that object as being in-use. If the object was already in-use User-A would be shown a change in the state of that object such as the object becomes disabled with some indication as to which user is using that object.  If that object was not in-use then User-A is flagged as using it and all other users who attempt to request access will be shown the change in state for that object.  This Client-Server architecture would require very little additional network traffic at run-time however it is not as dynamic as the architecture that allows for more traffic and greater feedback.

 

Self-Monitoring and Self-Tuning

 

The Client-Server Model could be self-tuning in terms of how often each Client polls for data.  As more Clients poll for data each Client could be made to sense this and increase the polling interval based on user feedback.  If User-A wanted to have a more dynamic experience the polling interval might be reduced to cause the Client to show changes more rapidly.  The Client could display a network activity indicator to show how many users are polling the servers in order to help the users have a smoother user experience without bogging down the servers too much.

 

Coding this type of self-monitoring and self-tuning Client-Server User Experience might require some additional development time but much of this effort would only need to be coded once.  One of the artifacts the process of coding this application model would be a set of sub-programs and routines that could be used to create additional applications with less effort and the first effort.

 

Benefits of Client-Server JavaScript

 

The benefit of having this type of dynamic self-monitoring and self-tuning Client-Server User Experience would be more multi-user functionality than is possible without this type of application framework.

 

Client-Server JavaScript Code Library

 

Many times during the development of a web application a programmer is called upon to create code that could have come from a Corporate Code Library.  A Corporate Code Library might contain code that knows how to populate the GUI with a menu of buttons.  This menu of buttons might have some known behaviors such as how the buttons are enabled and disabled.  The actual logic a button click fires-off could be supplied at run-time but the core functionality would only need to be coded once and then reused many times.

 

The next logical step beyond making a Corporate Code Library would be to create a Corporate Code Generator which is also known as an Application Generator.

Global Variables

 

There are two distinct types of Global Variables that are defined by this system.  ColdFusion Global Variables and JavaScript Global Variables.

ColdFusion Global Variables

 

ColdFusion Global Variables are primarily created in the application.cfm file however in real terms ColdFusion Global Variables can be created almost anywhere within any ColdFusion  template.

 

ColdFusion Global Variables Defined

 

ColdFusion Global Variables are defined as being any variable that is defined or created within a scope that allows the variable in question to exists as a Global with visibility from any ColdFusion template.

 

ColdFusion Global Variables Exceptions

 

The exception to the rule in terms of how and when Global Variables are defined or created in ColdFusion where this system is concerned is the ColdFusion Component (.CFC) file which is named “CommonCode.CFC”.   At the time of this writing, it was understood that variables that are created within the scope of a ColdFusion Component remain with this scope and are not visible outside this scope.  This is how ColdFusion Component(s) are used within this system however whether or not variables that are created or defined within a given ColdFusion Component are visible outside the scope of the ColdFusion Component has not been verified and no effort has been expected to determine if this is the case or not.  Likewise, variables created or defined within any ColdFusion Components are not typically used outside the scope in which they were created.

JavaScript Global Variables

 

JavaScript Global Variables are primarily created or defined within certain JavaScript Modules that are used all subsystems via the code reuse techniques employed by this system.

JavaScript Global Variables Defined

 

JavaScript Global Variables are defined as those variables that are defined or created outside the scope of any functions or procedures such as those that are created or defined at the top of the typical JavaScript Module file or those variables that are created or defined inline such as those that provide behaviors for HTML Events such as the “onclick” event or the like.

JavaScript Global Variables Exceptions

 

The exception to the rule for JavaScript Global Variables are those variables that typically exist as ColdFusion Global Variables that are also defined as JavaScript Global Variables.

 

If JavaScript Client-Server programming techniques had been used for this system it would not have been necessary to create a bridge between ColdFusion and JavaScript in terms of creating Global Variables that exist in two places.  JavaScript Client-Server programming techniques provides the means to directly manipulate data objects from a database directly in JavaScript through a ColdFusion Gateway Process.

 

Global Variables and JavaScript Client-Server Programming Techniques

 

JavaScript Client-Server programming techniques provide the means for directly manipulating data objects from a database directly within JavaScript without the need to manipulate or store those data objects in ColdFusion variables.

 

ColdFusion is used as a Gateway or conduit through which the data objects are transmitted directly to JavaScript for direct manipulation and processing.

 

JavaScript Client-Server programming techniques provides for a faster smoother user experience than the traditional ColdFusion programming techniques can offer.  For instance, there is no need to perform a “post” from an HTML Form to the ColdFusion server whenever it becomes necessary for users to interact with the database.  This means the user’s experience is not interrupted by having to wait for their browser page to refresh at the conclusion of the typical “post” operation.

 

JavaScript Client-Server programming techniques provides the means to maintain a much cleaner run-time environment because Global JavaScript Variables need only be defined in one .JS file which is loaded only once, when the page is refreshed.

Database Relationship Diagram Notes

 

There are a few simple rules used throughout the design process for the database for this system.

 

1.1.1.1.13       Database Relationship Notes

 

1). There are no relationships specified within the database itself.  The typical method employed for conveying the notion of “Relationships” in a Relational Database is to use a series of Primary_Keys[1] and/or Foreign_Key[2] Relationships however the development cycle for this system was quite short so time was of the essence and it was determined because of this as little time as possible was spent on each aspect of the system and the database was no exception.  Had the timeframe for the development phase been known in advance sufficient time could have been invested in performing a reasonable analysis to determine the database design however this was not the case and the database reflects this, for better or worse.

 

1.1.1.1.14       Central Table Notes

 

2). There is one central Table around which almost every other table has been anchored, namely the ReleaseManagement Table.  As you can see in the diagram below, there are only three groups of tables in this database.  Two groups have intra-group relationships and one inter-group relationship.  One group of tables have no relationships to the other two groups and this was done by-design.

 

1.1.1.1.15       Primary_Key Notes

 

3).  The releaseNumber column in the ReleaseManagement table is a Primary_Key however this was done to enforce uniqueness for the releaseNumber column because the releaseNumber needs to be a unique value in the database in order to maintain the meaning of this data entity.  In every single case where there is a stated Primary_Key attribute the goal was to cause the database engine to enforce uniqueness for that specific data entity.

 

1.1.1.1.16       Foreign_Key Notes for releaseNumber

 

4). The id column in the ReleaseManagement table is used as a Foreign_Key where the column name rid is used.  The columns named rid may be thought of as being the Release ID for a specific Release that has a humanly readable releaseNumber.

 

1.1.1.1.17       Foreign_Key Notes for uid

 

5).  The id column in the UserSecurity table is used as a Foreign_Key where the column name uid is used.  The columns named uid may be thought of as being the User ID for each instance of this data entity.

 

1.1.1.1.18       Foreign_Key Notes for pid

 

6).  The id column in the ContentList table is used as a Foreign_Key where the column name pid is used.  The columns named pid may be thought of as being the Page ID for each instance of this data entity.

 

1.1.1.1.19       Foreign_Key Notes for sid

 

7).  The id column in the SecurityList table is used as a Foreign_Key where the column name sid is used.  The columns named sid may be thought of as being the System ID for each instance of this data entity.

 

1.1.1.1.20       MarqueeScrollerData Table Notes

 

8).  The MarqueeScrollerData table is not directly associated with any other tables because there was insufficient time to fully develop the GUI for the Marquee Announcements Editor to allow for the need to be able to gather Marquee data from other Releases while working with a Draft Release.

 

1.1.1.1.21       DynamicLayoutSpecification Table Notes

 

9).  The DynamicLayoutSpecification table is an unused table using the current implementation of the system; this table was to be used with the GUI based Layout Subsystem before it was determined that this subsystem’s design as a GUI based component would take far too long to develop in favor of the easier to code Layout Spec Editor.

 

1.1.1.1.22       Redundant Content Table Notes

 

10).  The following tables could have been combined into a single database table had sufficient development time been allocated for this task:  DynamicHTMLContent, DynamicHTMLfooter, DynamicHTMLpad, DynamicHTMLright_side, DynamicHTMLsepg_links and DynamicHTMLsepg_section.  The choice was made to “firewall” the process of coding each of the content types in order to reduce the chance that previously written code might be broken by code written later on during the development process.
